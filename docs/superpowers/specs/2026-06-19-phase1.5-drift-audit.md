### 2. Quote — `passengers` Field

**Type file:** `src/types/quote.ts:199`  
**QuoteDraft field:** `passengers?: Passenger[]` — manifest + rooming list.

**Firebase persistence:** `passengers` is part of `QuoteDraft` and is saved wholesale as `currentState` inside `quote_projects/{cloudId}` via `fbSaveQuoteState` (firebase.ts:329–362).

**Schema:** No `quote_passengers` table in any of migrations 0007–0022. The `save_quote_state` RPC (0021) upserts quote columns and replaces line_items / flights / workflow / groups / payments — but `passengers` is not in the `INSERT INTO quotes(...)` column list and has no child table.

**Gateway:** `decomposeQuote` (quoteMap.ts:252–281) does not include `passengers` in the payload. `assembleQuote` (quoteMap.ts:71–250) does not reconstruct it. **The field is completely absent from the Supabase path.**

**However** — `passengers` is stored inside `quote_versions.state` (JSONB full snapshot) and is restored from there when loading a version. So **version-based round-trips preserve it**, but the `currentState` columns and the `quotes` row have no direct storage for it. On `sbGetQuoteProject`, `assembleQuote` returns a `QuoteDraft` without `passengers` even if the live quote has passengers. Only `sbSaveQuoteState`/`sbGetQuoteProject` on the version `state` JSONB preserves passengers.

**Severity:** data-loss-on-write for `currentState`. The working passenger list is not queryable and will be missing from the assembled currentState returned by `sbGetQuoteProject`.

**What's needed:**
- Schema: add `quote_passengers` table (id uuid, quote_id uuid FK, legacy_passenger_id text, name text, gender text, dob text, id_type text, id_no text, nationality text, room_type text, room_no text, dietary text, phone text, emergency text, note text, sort_order int).
- quoteMap.ts: add `passengers` to `decomposeQuote` and `assembleQuote`.
- RPC 0021: add `passengers` delete + re-insert block.

### 3. Quote — `WorkflowStep.attachments` Field

**Type file:** `src/types/quote.ts:177`  
**WorkflowStep field:** `attachments?: FileAttachment[]` — per-step file attachments stored on R2.

**Schema:** `quote_workflow_steps` table (migration 0009) has: id, quote_id, legacy_step_id, label, status, step_key, due_offset, start_date, due_date, done_date, assignee_user_id, assignee_username, note, sort_order. No attachments column or separate attachment table for workflow steps.

**Gateway:** `workflowRows` in quoteMap.ts:35–41 maps WorkflowStep → row, omitting `attachments`. `assembleQuote` step reconstruction (lines 151–169) does not include `attachments`. The `attachments` table is used generically (parent_type/parent_id) but no code calls `saveAttachments(client, 'quote_workflow_step', ...)`.

**Like `passengers`**, step attachments are preserved inside `quote_versions.state` JSONB snapshots but not in the live `currentState` assembled from relational rows.

**Severity:** data-loss-on-write for currentState. Step attachments are stripped from `assembleQuote` output.

**What's needed:**
- Use the existing generic `attachments` table with `parent_type='quote_workflow_step'` and `parent_id=legacy_step_id`.
- quoteMap.ts: add attachment save/load for each workflow step in `decomposeQuote` / `assembleQuote`.
- RPC: call `saveAttachments` per step (RPC is SQL; step attachment save would need to happen in the gateway layer after the RPC, similar to how `saveAttachments(client, 'quote', ...)` is called in `saveSingleQuoteEntry`).

**Type file:** `src/types/itinerary.ts:93`

### 6. Chat — Entire Feature Uncovered

**Type file:** `src/types/chat.ts`

#### Full Chat Data Model

**`Chat` type** (`src/types/chat.ts:21–34`):
```
Chat {
  id: string                        // 'dm_{a}__{b}' for 1-1; uuid for group
  members: string[]                 // usernames
  isGroup: boolean
  title?: string                    // group name
  createdBy: string                 // username
  createdAt: string                 // ISO
  messages: ChatMessage[]           // capped at 500 in fb*
  lastAt?: string                   // ISO — last message timestamp
  lastText?: string                 // preview of last message
  lastByName?: string
  reads?: Record<string, string>    // username → ISO (last-read per member)
}
```

**`ChatMessage` type** (`src/types/chat.ts:7–18`):
```
ChatMessage {
  id: string
  by: string                        // username
  byName: string
  at: string                        // ISO
  text?: string
  file?: ChatFile                   // { key, name, size, mime? }
  replyTo?: ChatReply               // { id, byName, text }
  editedAt?: string                 // ISO
  deleted?: boolean
  reactions?: Record<string, string[]>  // emoji → username[]
}
```

**`ChatFile` type** (`src/types/chat.ts:2`): `{ key: string; name: string; size: number; mime?: string }` — R2 reference.

**Operations supported by fb* layer:**
- `fbSubscribeChats(username)` — realtime list of chats where `members array-contains username`
- `fbSubscribeChat(id)` — single chat realtime
- `fbEnsureChat(chat)` — create or merge-members
- `fbSendChatMessage(id, msg)` — read-modify-write; updates `lastAt`/`lastText`/`lastByName`/`reads`
- `fbEditChatMessage(id, msgId, text)` — edit text + `editedAt`
- `fbDeleteChatMessage(id, msgId)` — soft-delete (sets `deleted: true`, clears text/file)
- `fbToggleChatReaction(id, msgId, emoji, username)` — toggle emoji reaction
- `fbMarkChatRead(id, username)` — update `reads[username]`

#### Proposed Tables for Supabase Migration

```sql
create table public.chats (
  id          text primary key,                   -- 'dm_{a}__{b}' or uuid
  is_group    boolean not null default false,
  title       text,
  created_by  uuid references public.profiles(id),
  created_by_name text,
  created_at  timestamptz not null default now(),
  last_at     timestamptz,
  last_text   text,
  last_by_name text
);

create table public.chat_members (
  chat_id   text not null references public.chats(id) on delete cascade,
  user_id   uuid references public.profiles(id),
  username  text not null,
  last_read timestamptz,
  primary key (chat_id, username)
);

create table public.chat_messages (
  id          uuid primary key default gen_random_uuid(),
  chat_id     text not null references public.chats(id) on delete cascade,
  legacy_id   text,                                -- ChatMessage.id (client-generated)
  by_user_id  uuid references public.profiles(id),
  by_username text not null,
  by_name     text not null default '',
  at          timestamptz not null default now(),
  text        text,
  file        jsonb,                               -- ChatFile: {key,name,size,mime?}
  reply_to    jsonb,                               -- ChatReply: {id,byName,text}
  edited_at   timestamptz,
  deleted     boolean not null default false,
  reactions   jsonb not null default '{}'::jsonb,  -- emoji → username[]
  sort_order  bigint generated always as identity  -- for stable ordering
);
create index chat_messages_chat_idx on public.chat_messages(chat_id);
create index chat_messages_at_idx on public.chat_messages(at);
```

**Realtime note:** `chats` table needs to be added to the Realtime publication (migration 0016 pattern). Use `chat_id=eq.{id}` filter for per-chat subscriptions; use `chats` table subscription filtered by members for the list (or use a user-specific materialized view).

#### Proposed `sb*` Gateway Functions

| Function | Mirrors | Notes |
|----------|---------|-------|
| `sbSubscribeChats(username, cb)` | `fbSubscribeChats` | Query `chat_members` for username, join `chats` |
| `sbSubscribeChat(id, cb)` | `fbSubscribeChat` | Subscribe to `chat_messages` for chat_id |
| `sbEnsureChat(chat)` | `fbEnsureChat` | Upsert `chats`, merge into `chat_members` |
| `sbSendChatMessage(id, msg)` | `fbSendChatMessage` | Insert into `chat_messages`; update `chats.last_*`; update `chat_members.last_read` for sender |
| `sbEditChatMessage(id, msgId, text)` | `fbEditChatMessage` | UPDATE chat_messages set text, edited_at |
| `sbDeleteChatMessage(id, msgId)` | `fbDeleteChatMessage` | UPDATE chat_messages set deleted=true, text=null, file=null |
| `sbToggleChatReaction(id, msgId, emoji, username)` | `fbToggleChatReaction` | UPDATE chat_messages.reactions jsonb |
| `sbMarkChatRead(id, username)` | `fbMarkChatRead` | UPDATE chat_members.last_read |

**Unread count helper:** Replace `chatUnread` (chatStore.ts:22–28) with a query on `chat_members.last_read` vs `chat_messages.at` for the given user.

## Recommended Phase 1.5 Scope

Phase 1.5 should resolve all **data-loss-on-write** gaps before any cutover. In priority order:

### Must-fix before cutover (data-loss)

1. **Customer CRM fields** (source, tags, interactions, nextFollowUp)
   - Migration: add 3 columns to `customers` + new `customer_interactions` table.
   - Gateway: update `sbPushCustomers` and `rowToCustomer`.

2. **Notification priority, attachments, reminder**
   - Migration: add `priority text`, `reminder jsonb` to `notifications`.
   - Gateway: update `sbSendNotification`, `sbPushNotifications`, `rowToNotif`. Add attachment save/load.

3. **Quote passengers**
   - Migration: new `quote_passengers` table.
   - quoteMap.ts + RPC: add passengers decompose/assemble block.

4. **Quote WorkflowStep attachments**
   - Use existing `attachments` table with `parent_type='quote_workflow_step'`.
   - quoteMap.ts: add attachment save/load per step in decompose/assemble (post-RPC in gateway layer).

5. **Itinerary startDate**
   - Migration: `alter table public.itineraries add column if not exists start_date date`.
   - Gateway: add to `sbSaveItinerary` upsert payload and `sbGetItinerary` assembly.

### Should-fix (whole feature gap)

6. **Chat feature** (§6 above)
   - 3 new tables: `chats`, `chat_members`, `chat_messages`.
   - Realtime publication entry.
   - 8 `sb*` gateway functions mirroring all `fb*` chat functions.
   - Update `chatStore.ts` to use `sb*` functions when Supabase mode is active.

### Nice-to-fix (cosmetic)

7. **WorkflowStep assignee_user_id** — populate UUID FK during save (low impact).

# PR-3.7 PaymentView — Design

_Date: 2026-06-08_
_Phase: 3 (tab-by-tab port) — Per-tour payment tracker_
_Builds on: PR-3.6 Contract (`fd2b5df`), PR-3.9 DMC (`66f2470`)_
_Status: Approved design, ready for implementation plan_

---

## 1. Goal

Port the **per-tour payment tracker** (legacy `PaymentView`, `public/legacy.html:4311–4675`) into the Vite app as a sub-view inside the **Quote tab** (alongside `cost`, `summary`, `dashboard`, `history`). After this PR a sales user can:

- Switch to **Thanh toán** from the in-Quote view toggle.
- See **4 summary cards**: tổng chi phí quản lý / đã thanh toán / công nợ còn thiếu / đã lên lịch.
- See cost items **grouped by category** with editable supplier, override amount, expandable installments (label + amount + paid date + paid toggle), and a per-item progress ring.
- **Pick which items to track** via the "Quản lý hạng mục" modal.
- **Add custom cost items** (chi phí tự tạo, not in the rate-card breakdown) via the "Thêm chi phí tự tạo" modal.
- Have payment data **persisted per-tour** in localStorage and synced to Firestore so collaborators see updates in real time.

**Approval flow** — included in this PR (legacy 2-stage flow). After build:
- "📄 Phiếu ĐN" button on each installment opens `PaymentRequestModal` with an approver-1 / approver-2 picker (filtered to `CEO` / `Trưởng Phòng`), supplier, content, amount, note.
- "📄 Xuất PDF" generates a Vietnamese "Phiếu đề nghị thanh toán" using `jspdf` (Helvetica only, no logo — same pattern as `exportContractPDF`).
- "📤 Gửi đề nghị" dispatches a `payment_approval` notification of the tour-payment shape (`{tourKey, ciKey, instIdx, approvalKey, approvalStage:1, approver1Username, approver1Name, approver2Username, approver2Name}`).
- `NotificationBell` is extended to detect the tour-payment notification shape (presence of `approvalKey`) and routes through the legacy 2-stage progression: stage-1 approve → write `fbSetApprovalStage(key,1,'approved',…)` → send stage-2 notification to approver2; stage-2 approve → write `fbSetApprovalStage(key,2,'approved',…)` → mark installment `status='paid'` + `paidDate=today` on the tour-payments doc. Reject at any stage → write rejection + notify requester.
- Each installment row in `PaymentView` shows a 3-state badge (`⏳ Chờ duyệt` / `✅ Đã duyệt` / `❌ Từ chối`) driven by a Firestore subscription on `viettours/payment_approvals`.
- Re-assignment: clicking 📄 Phiếu ĐN on an installment that already has approval state opens the modal pre-filled with the stored stage-1/stage-2 names, and the send-button label switches to "🔄 Cập nhật người duyệt".

**Out of scope (deferred — listed under refactor spec §8 / `CONTEXT.md` "Next Steps"):**
- Notification-tab full-page view (CONTEXT.md "Next Steps" item 3).
- Acceptance deadline reminders (item 4).
- Mobile responsiveness (item 5).
- Tests — refactor convention.

DMC template **does not** get a payment view (legacy hides it; consistent with PR-3.9 §V8).

## 2. Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Where state lives | **New `paymentStore` keyed by `tourKey`** (separate from `quoteStore.draft`) | Legacy keys payments by `info.name` → `tourKey`, shared across every quote/version of the same tour. Storing inside the draft would force per-version snapshots and break the legacy collaboration model. |
| Store granularity | One store, one Firestore subscription, in-memory map `{ [tourKey]: { payments, customItems } }`. Subscribes lazily — only when PaymentView mounts and `info.name` is non-empty. | Subscribing to every tour at app start is wasteful; legacy already subscribes lazily on view mount. |
| View slot | Extend `QuoteState.view` union with `'payment'`. Toolbar gets a `Thanh toán` toggle button, hidden for DMC. `QuoteView.tsx` routes the new value. | One-line union change matches the pattern PR-3.9 used for clamp logic. |
| DMC clamp | Clamp `view: 'payment'` → `'cost'` when `template === 'dmc'` in `setView` and `init` (extend the existing clamp added in PR-3.9). | Same defensive clamp as DMC; prevents stale state after template switch. |
| Cost-item source | Recompute `sourceItems` on the fly from `draft.items`, `draft.catEnabled`, `draft.pax`, `draft.rates`, and the template's active categories — identical to legacy `4320-4336`. | Avoids duplicating cost data; auto-tracks edits made on the Cost view. |
| Storage location | localStorage keys `vte_payments_{tourKey}` + `vte_pay_custom_{tourKey}` mirror legacy verbatim. Firestore doc `tour_payments/{tourKey}` mirrors legacy `_tpDoc`. | Two-app coexistence: legacy and Vite write the same docs/keys until cleanup. |
| Firestore signatures | `fbSaveTourPayments(tourKey, payments, customItems)` and `fbSubscribeTourPayments(tourKey, cb)` exported from `src/lib/firebase.ts`. Mirror legacy `window.fbSaveTourPayments` / `window.fbOnTourPayments` exactly. | Mechanical port per refactor spec §4.1. |
| Debounce | 1s debounce on Firestore writes per `tourKey`, identical to legacy `4352-4359`. localStorage writes are immediate. | Avoids write storms on rapid edits (e.g. typing in supplier name). |
| Cross-tab sync | `onSnapshot` is the only real-time channel. No `storage` event listener needed (legacy doesn't have one either). | Keep parity. |
| `tourKey` derivation | `slugifyTourKey(info.name)` = `info.name.replace(/[^a-zA-Z0-9]/g, '_')`. Empty `info.name` → `'tour'` fallback, same as legacy. | Verbatim copy. |
| Empty-name guard | If `info.name` is blank, show an info Alert "Đặt tên tour ở mục Thông tin trước khi quản lý thanh toán" and skip the subscription. | Prevents collisions on the legacy `'tour'` fallback (multiple unnamed quotes would share state). |
| Currency display | VND only (no DMC currency conversion). | Legacy renders VND only on this view; DMC has no payment view at all. |
| Number-input control | Reuse `EditableNumber` from `src/components/quote/EditableNumber.tsx` (same component used by `LineRow`). | Established pattern; auto VND suffix support. |
| Modals styling | MUI `Dialog` + custom header strip (gradient teal / amber) to preserve legacy visual identity, consistent with how `ContractModal` / `SaveCloudQuoteModal` were ported. | Visual continuity. |
| Permission gates | None — PaymentView is accessible whenever the Cost view is. Approval-only actions (out of scope) would have had gates. | Matches legacy: no gate on the view itself. |
| Legacy migration | On store `init()`, scan `localStorage` for `vte_payments_*` / `vte_pay_custom_*` keys and hydrate the in-memory map. Do **not** delete legacy keys — legacy app is still served at `/legacy.html` and may write them. | Cleanup is a Phase 4 concern, not this PR. |

## 5. Firestore signatures — `src/lib/firebase.ts`

Add at the bottom of the file (after the DMC quote-history family):

```ts
const _tpDoc = (tourKey: string) => doc(db, 'tour_payments', tourKey);

export async function fbSaveTourPayments(
  tourKey: string,
  payments: Record<string, PaymentRecord>,
  customItems: CustomCostItem[],
): Promise<void> {
  const updatedBy = useAuthStore.getState().currentUser?.name ?? 'unknown';
  await setDoc(_tpDoc(tourKey), {
    payments,
    customItems: customItems ?? [],
    updatedAt: new Date().toISOString(),
    updatedBy,
  });
}

export function fbSubscribeTourPayments(
  tourKey: string,
  cb: (data: TourPayments | null) => void,
): Unsubscribe {
  return onSnapshot(_tpDoc(tourKey), (s) => {
    if (!s.exists()) return cb(null);
    const data = s.data();
    cb({
      payments: (data.payments ?? {}) as Record<string, PaymentRecord>,
      customItems: (data.customItems ?? []) as CustomCostItem[],
    });
  });
}
```

## 6. Store — `src/stores/paymentStore.ts`

In-memory map keyed by `tourKey`. Lazy subscribe on `ensureSubscribed(tourKey)`; reference-counted dispose via `releaseSubscription(tourKey)` so re-mounts don't tear down and re-fetch.

```ts
type PaymentSlot = {
  data: TourPayments;
  unsub: Unsubscribe | null;
  refCount: number;
  pushTimer: ReturnType<typeof setTimeout> | null;
};

type PaymentState = {
  slots: Record<string, PaymentSlot>;
  init: () => void;                                    // hydrate from localStorage
  ensureSubscribed: (tourKey: string) => void;
  releaseSubscription: (tourKey: string) => void;
  setPayments: (tourKey: string, next: Record<string, PaymentRecord>) => void;
  setCustomItems: (tourKey: string, next: CustomCostItem[]) => void;
  getTour: (tourKey: string) => TourPayments;          // selector helper
};
```

Behaviors:

- `init()` — scans `localStorage` for `vte_payments_*` and `vte_pay_custom_*` keys, populates `slots[tourKey].data`. No Firestore reads; subscriptions happen on demand.
- `ensureSubscribed(tourKey)` — if `slot.unsub` is null, `fbSubscribeTourPayments` and store the returned unsub. Increment `refCount`. On each cloud update, write to slot + mirror to localStorage.
- `releaseSubscription(tourKey)` — decrement; when `refCount` hits 0 call `unsub()` and null it out. (Slot data stays in memory; only the subscription is torn down.)
- `setPayments` / `setCustomItems` — update slot, write localStorage immediately, clear-and-set a 1s `pushTimer` that calls `fbSaveTourPayments` with the latest state.

PaymentView wires `ensureSubscribed` in a `useEffect`, returning `releaseSubscription` for cleanup.

## 10. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Two-app dual-write conflicts (Vite + legacy each push to `tour_payments/{tourKey}`) | Low | Last-write-wins is acceptable here — payments aren't operationally critical mid-session. `updatedAt` field carries the timestamp for forensic review. |
| Slug collision when `info.name` is empty (all unnamed quotes share `tour_payments/tour`) | Medium | Empty-name guard (§2 + §8) blocks subscription and edits until a name is set. |
| Re-render storms on large `payments` map (many items × installments) | Low | PaymentView already groups by category; React renders only mounted accordion bodies (expand state). No memo needed initially; revisit if perf complaints. |
| Per-item supplier input writes on every keystroke debounced 1s — pre-debounce edits are lost on tab switch | Low | localStorage write is immediate (not debounced); only Firestore push is debounced. Tab switch + reload restores from localStorage. |
| Cloud subscription fires shortly after our own debounced push, overwriting unrelated local edits | Low | We write the whole document in one call and the subscription delivers the same payload back. Race with another user is possible but legacy has the same race. |
| `tourKey` change while subscription is alive (user renames tour info) | Medium | `ensureSubscribed(newKey)` + `releaseSubscription(oldKey)` triggered by `useEffect` dependency on `tourKey`. Cleanup runs before the next mount per React semantics. |
| Custom cost items with `amount` shape divergence from `EditableNumber` parsing | Low | Reuse the same control used in `LineRow`, which already round-trips numbers correctly. |

## 10b. Approval flow — additional details

### 10b.1 Firestore signatures (add to `src/lib/firebase.ts`)

```ts
import type { PaymentApprovalDoc, PaymentApprovalStage } from '@/types';

const _PA_DOC = doc(db, 'viettours', 'payment_approvals');

export async function fbSetApprovalStage(
  key: string,
  stage: 1 | 2,
  status: 'approved' | 'rejected',
  approverUsername: string,
  approverName: string,
  note: string,
  intended: { intendedApprover1Name?: string; intendedApprover2Name?: string } = {},
): Promise<void> {
  const snap = await getDoc(_PA_DOC);
  const ex: PaymentApprovalDoc = snap.exists() ? (snap.data() as PaymentApprovalDoc) : {};
  const existing = ex[key] ?? {};
  const stageData: PaymentApprovalStage = {
    status,
    approverUsername,
    approverName,
    note,
    updatedAt: new Date().toISOString(),
  };
  const finalStatus =
    status === 'rejected' ? 'rejected' : stage === 2 ? 'approved' : 'pending_stage2';
  ex[key] = {
    ...existing,
    [`stage${stage}`]: stageData,
    currentStage: stage,
    finalStatus,
    ...(intended.intendedApprover1Name ? { intendedApprover1Name: intended.intendedApprover1Name } : {}),
    ...(intended.intendedApprover2Name ? { intendedApprover2Name: intended.intendedApprover2Name } : {}),
  };
  await setDoc(_PA_DOC, ex);
}

export function fbSubscribePaymentApprovals(
  cb: (data: PaymentApprovalDoc) => void,
): Unsubscribe {
  return onSnapshot(_PA_DOC, (s) => cb(s.exists() ? (s.data() as PaymentApprovalDoc) : {}));
}
```

Mirror legacy `public/legacy.html:390-412`. We **skip** the legacy single-stage `fbSetPaymentApproval` (deprecated; only used for backward-compat reads in legacy).

### 10b.4 Notification routing (`NotificationBell.tsx` edit)

Detection: `notif.type === 'payment_approval' && notif.data?.approvalKey != null` → tour-payment shape. Otherwise → existing contract-payment shape.

New handler `handleTourPaymentApproval(approved)`:

1. Resolve `data` as `TourPaymentApprovalData`.
2. `await fbSetApprovalStage(data.approvalKey, data.approvalStage, approved ? 'approved' : 'rejected', currentUser.u, currentUserName + ' (' + currentUser.role + ')', '', { intendedApprover1Name: data.approver1Name, intendedApprover2Name: data.approver2Name })`.
3. If `approved && data.approvalStage === 1` and `data.approver2Username` is present → send a stage-2 notification (same shape, `approvalStage: 2`) to `data.approver2Username`.
4. If `approved && data.approvalStage === 2` (final approval) → mark the installment paid: `paymentStore.setPayments(data.tourKey, …)` where `payments[data.ciKey].installments[data.instIdx]` gets `status: 'paid', paidDate: todayISO()`. Re-uses the lazy `ensureSubscribed`/`releaseSubscription` lifecycle inside a one-shot helper that does ensure → setPayments → release.
5. Send reply notification to `data.requestedBy` ("✅ Đã được duyệt" / "❌ Bị từ chối").
6. Mark current notification read.

The contract-payment handler stays unchanged.

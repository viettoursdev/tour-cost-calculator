## 1. Goal

Port the **Hợp Đồng (Contract) tab** — CRUD + payment tracking workflow — into the Vite app. This is PR-3.6 (ContractView + ContractModal + PaymentPanel + AcceptanceCertModal). **ContractManagerModal** (the 3,000-line detail viewer + PDF/Word exporter) is deferred to PR-3.7.

After this PR: Sales/Ops can create, edit, delete, and track payments for contracts directly from the Vite app. PDF/Word export still uses `/legacy.html`.

## 3. Decisions

| Decision | Choice | Rationale |
|---|---|---|
| ContractManagerModal | **Deferred to PR-3.7** | 3,000-line component; tightly coupled to PDF/Word export (Phase 4). Porting it alone ships no value without the exports. |
| Storage | **Firestore only** (`viettours/contracts_master.contracts[]`) | No localStorage migration. Assume Firestore is source of truth. |
| ContractModal tour-info pre-fill | **Quote picker from `quoteHistoryStore.visibleQuotes()`** + "Thêm trống" fallback | Sales always creates contracts from saved quotes. Avoids cross-store tight coupling — reads at open time only. |
| ContractModal form | **Full port** (all fields: partyB, includes/excludes, payment schedule, cancellation policy, bond%) | Operational necessity; simplified version would require legacy for most real workflows. |
| Payment approval notifications | **Deferred to PR-3.10** | `fbSendNotification` flow; separate notification PR. PaymentPanel renders payment rows without the approval request button. |
| Acceptance cert PDF export | **Deferred to Phase 4** | Export functions. AcceptanceCertModal still records the cert date/note in Firestore. |
| Permission model | `manageContracts` → full CRUD; `viewContracts` (Admin) → read-only list | Matches legacy `canEdit`/`viewOnly` flags. |
| Tests | **Out of scope** | Refactor convention. |

## 6. Firestore API (`src/lib/firebase.ts` additions)

```ts
const CONTRACTS_DOC = doc(db, 'viettours', 'contracts_master');

export function fbSubscribeContracts(cb: (list: Contract[]) => void): Unsubscribe {
  return onSnapshot(CONTRACTS_DOC, (snap) => {
    cb(snap.exists() ? ((snap.data().contracts as Contract[]) ?? []) : []);
  });
}

export async function fbGetContracts(): Promise<Contract[]> {
  const snap = await getDoc(CONTRACTS_DOC);
  return snap.exists() ? ((snap.data().contracts as Contract[]) ?? []) : [];
}

export async function fbPushContracts(
  list: Contract[],
  pushedBy: { name: string; role: string },
): Promise<void> {
  await setDoc(CONTRACTS_DOC, {
    contracts: list,
    updatedAt: new Date().toISOString(),
    updatedBy: `${pushedBy.name} (${pushedBy.role})`,
  });
}
```

## 7. `contractStore`

```ts
type ContractState = {
  contracts: Contract[];
  loading: boolean;
  syncing: boolean;
  init: () => Unsubscribe;
  save: (form: Contract) => Promise<void>;
  delete: (id: string) => Promise<void>;
  updatePayments: (id: string, payments: ContractPayment[]) => Promise<void>;
  markAcceptance: (id: string, date: string, note: string) => Promise<void>;
  updateStatus: (id: string, status: Contract['contractStatus']) => Promise<void>;
};
```

Zustand `subscribeWithSelector`, no `persist`. All write actions: optimistic set → `fbPushContracts` → `syncing: false`. Same pattern as NCC/Customer.

`save(form)`:
- If `contracts.find(c => c.id === form.id)` exists → update: add `updatedAt/updatedBy`.
- Else → create: generate `id = 'hd_' + Date.now()`, add `createdAt/createdBy`.
- Recalculate payment amounts from percent if `percent` is set: `amount = Math.round(totalAmount * percent / 100)` where `totalAmount = pricePerPax * contractPax`.

`updatePayments(id, payments)`: patch single contract's payment array → push.

`markAcceptance(id, date, note)`: sets `hasAcceptance: true, acceptanceDate, acceptanceNote, contractStatus: 'completed'` → push.

`updateStatus(id, status)`: patches `contractStatus` → push.

## 8. Component tree

```
<ContractView>
  ├─ Stats row: total contract value · total paid · count per status
  ├─ Search TextField + status Select filter + [✕ clear]
  ├─ [➕ Thêm hợp đồng] Button (if manageContracts)
  │
  ├─ Loading LinearProgress
  ├─ Empty state
  └─ Contract list (MUI Accordion, one per contract)
       ├─ AccordionSummary:
       │    contractNo · tourName · partyB.name · {pax} khách
       │    status Chip · totalAmount fmtVND
       │    [inline status Select] (if canEdit)
       │    [✏️ edit] [🗑️ delete] IconButtons (if canEdit)
       └─ AccordionDetails:
            └─ PaymentPanel (receives contract as prop)
               + [📋 Biên bản nghiệm thu] Button → AcceptanceCertModal
                 (shown if contractStatus==='completed' && canEdit, OR hasAcceptance)

{quotePicker → <QuotePickerDialog/>}
{createModal || editModal → <ContractModal/>}
{acceptanceTarget → <AcceptanceCertModal/>}
{deleteTarget → <ConfirmDeleteDialog/>}
```

### `QuotePickerDialog`

Small MUI `<Dialog>` listing `useQuoteHistoryStore(s => s.visibleQuotes())`. Each `<ListItemButton>` shows: quote name, destination, pax, totalCost. Two actions: "Chọn" per row + "Thêm trống" in `DialogActions`. Closes and passes the selected quote (or null for blank) to the caller.

### `ContractModal`

`<Dialog fullWidth maxWidth="md">` with MUI `<Tabs>` for 4 sections:

| Tab | Fields |
|---|---|
| Thông tin chung | contractNo, contractDate (date picker), contractStatus (Select), tourName, tourDest, tourDays, tourNights, tourStartDate, departure, contractPax, pricePerPax → totalAmount (computed, readonly), bondPercent; **partyB** subform: name, address, tel, rep, title, taxCode, email |
| Dịch vụ | includes[] editable list (TextField per item + add/remove); excludes[] same |
| Thanh toán | payments[] list: label, percent (optional) or fixed amount, dueDate, note; add/remove rows; totalAmount shown at top |
| Phạt huỷ | cancels[] list: when (text), penalty (number%); add/remove rows |

Save button: disabled when `form.tourName` or `form.partyB.name` is empty.

### `PaymentPanel`

Receives `contract` as prop (not subscribed to store). Calls `contractStore.updatePayments(contract.id, next)` on mutations.

- Progress `<LinearProgress>` showing `totalPaid / totalAmount`.
- Per payment `<Paper>`: label · amount (VND) · dueDate · status chip · [✓ Đánh dấu đã TT] toggle button · received-amount inline edit · [✕] delete.
- [+ Thêm đợt] form: label, amount, dueDate, note.
- **No payment approval request button** (deferred to PR-3.10).

### `AcceptanceCertModal`

Simple `<Dialog>` with: date `<TextField type="date">` + note `<TextField multiline>` + [Xác nhận nghiệm thu] button. On confirm: `contractStore.markAcceptance(id, date, note)`. **No PDF export button** (deferred to Phase 4).

## 10. Risks

| Risk | Mitigation |
|---|---|
| `contracts_master` doc size (many contracts × payment history) | Same single-doc pattern as legacy. Monitor at 500+ contracts. |
| `pricePerPax` pre-fill from quote: `totalCost / pax` may not be a clean integer | `Math.round(totalCost / pax / 1000) * 1000` — round to nearest 1,000 VND. User can override. |
| Payment `percent`-based amounts become stale if `pricePerPax` changes | Recalculate on every `save(form)` call. Same as legacy. |
| Legacy `_tourKey` field on contracts — Vite ignores it | Preserved via `{ ...existingContract, ...updates }` spread. |
| ContractModal 4-tab form is large (706 lines legacy) | Split is logical; each tab is self-contained. MUI Tabs handle it cleanly. |

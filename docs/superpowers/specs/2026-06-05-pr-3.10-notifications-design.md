# PR-3.10 Notifications — Design

_Date: 2026-06-05_
_Phase: 3 (cross-cutting feature)_
_Builds on: PR-3.6 (contracts), PR-3.2 (cloud history)_
_Status: Approved design, ready for implementation plan_

---

## 1. Goal

Port the notification system from the legacy app — in-app bell icon with dropdown, payment approval request/response flow, payment deadline reminders on login, and collaborator invite notifications.

After this PR: Sales can send payment approval requests to CEO/Trưởng Phòng directly from the Vite app. Managers see them in the notification bell, can approve or reject, and the contract payment status updates automatically. Users also receive deadline reminders on login for payments due within 7 days.

---

## 2. Firestore schema

**Path:** `user_notifications/{username}` — per-user document with a `notifications: Notification[]` array. Cap at 100 entries (FIFO). Public rules apply (same as all other collections).

**Notification shape:**

```ts
type NotificationType = 'payment_due' | 'payment_approval' | 'collab_invite';

type Notification = {
  id: string;            // Date.now().toString(36) at write time
  type: NotificationType;
  title: string;
  message: string;
  createdBy: string;     // user name or "Hệ thống"
  createdAt: string;     // ISO
  read: boolean;
  data?: Record<string, unknown>;  // type-specific: see below
};
```

**`data` shapes by type:**

| type | data fields |
|---|---|
| `payment_due` | `contractId, paymentId` |
| `payment_approval` | `contractId, contractNo, paymentId, paymentLabel, amount, requestedBy (username), requestedByName, assignedTo (username), assignedToName` |
| `collab_invite` | `quoteId, cloudId, quoteName, quoteCode, invitedBy` |

## 4. `notificationStore`

```ts
type NotificationState = {
  notifications: Notification[];
  unreadCount: number;          // computed on every set
  init: (username: string) => Unsubscribe;
  markAllRead: (username: string) => Promise<void>;
  markRead: (username: string, id: string) => Promise<void>;
};
```

- `init(username)` subscribes via `fbSubscribeNotifications`.
- `unreadCount` is `notifications.filter(n => !n.read).length`, computed on every `set`.
- `markAllRead` / `markRead`: flip `read: true` → `fbPushNotifications`.
- No `persist` — Firestore is the source of truth.

## 5. `checkContractDeadlines`

```ts
// src/lib/notifications.ts (new function — existing file already has
// requestBrowserNotifPermission and showBrowserNotif)
export async function checkContractDeadlines(user: User): Promise<void>;
```

Called from `MainApp.tsx` with a 3-second delay after login.

**Logic (ported verbatim from `legacy.html:10561-10585`):**
- Pull all contracts via `fbGetContracts()`.
- For each contract, iterate `payments`. If `status === 'pending'` and `dueDate` is within the next 7 days: send a `payment_due` notification to `user.u` via `fbSendNotification`.
- Catches and warns on failure — does not re-throw.

**Note:** This will send a new notification on every login if the payment is still pending and within 7 days. Legacy had the same behavior — no deduplication. Users see repeat reminders until they mark the payment as paid.

### `NotificationItem` (inline sub-component within `NotificationBell.tsx`)

```
<Paper sx={{ borderLeft: '4px solid {typeColor}', bgcolor: read ? 'inherit' : 'action.hover' }}>
  <Typography fontWeight={700}>{title}</Typography>
  <Typography variant="caption">{message}</Typography>
  <Typography variant="caption" color="text.secondary">{relative time}</Typography>

  {/* For payment_approval type, if current user is CEO or Trưởng Phòng */}
  {type === 'payment_approval' && canApprove && !alreadyActed && (
    <Stack direction="row" spacing={1}>
      <Button color="success" onClick={() => handleApproval(n, true)}>✅ Duyệt</Button>
      <Button color="error"   onClick={() => handleApproval(n, false)}>❌ Từ chối</Button>
    </Stack>
  )}
</Paper>
```

**Type colors:**
- `payment_due` → orange (`#f39c12`)
- `payment_approval` → teal (`#14A08C`)
- `collab_invite` → blue (`#2980b9`)

**`handleApproval(notif, approved)`:**
1. Mark the notification as `read: true` locally + push.
2. If `approved`: call `contractStore.updatePayments(data.contractId, updatedPayments)` — sets the target payment's `status: 'paid'`, `paidDate: today`.
3. Send a reply notification to `data.requestedBy` via `fbSendNotification` with type `payment_approval` and `approved: true/false`.
4. Update the notification's internal state to `acted: true` (stored in store only — does NOT write a new field to Firestore; the `acted` state is derived from whether a reply notification was sent within the session).

**Simplification vs legacy:** Legacy had 2-stage approvals (`approver1` + `approver2`). PR-3.10 implements single-approver only. The 2-stage path is deferred.

## 7. PaymentPanel: "Gửi đề nghị" button

Update `src/components/contract/PaymentPanel.tsx`. Add a new prop `currentUser: User | null`. For each pending payment where `!p.approvalRequested`:

- If `currentUser.role` is NOT `CEO` or `Trưởng Phòng` (i.e., Sales/Operations want approval):
  - Show `[📨 Gửi đề nghị duyệt]` button.
  - Clicking opens a small `<Dialog>` listing users with `role === 'CEO' || role === 'Trưởng Phòng'` from `authStore.users`.
  - Selecting an approver: calls `fbSendNotification(approver.u, { type: 'payment_approval', ... })` and marks `payment.approvalRequested = true` in local state → `contractStore.updatePayments`.

- If `p.approvalRequested && !p.status === 'paid'`: show `[⏳ Đã gửi đề nghị]` (disabled).

**`ContractView` update:** Pass `currentUser` down to `<PaymentPanel contract={c} canEdit={canEdit} onUpdate={...} currentUser={currentUser} />`.

## 10. Risks

| Risk | Mitigation |
|---|---|
| Duplicate deadline reminders on repeated logins | Accepted — same as legacy. Users see it until they mark the payment paid. |
| Race condition: two managers approve simultaneously | Last-write-wins on `contractStore.updatePayments`. The second approval overwrites the first. Acceptable for internal tool. |
| `acted` state lost on page refresh | The approval-acted state is session-only — buttons reappear on refresh. Managers could theoretically approve twice. Mitigation: the contract payment status will already be `paid` from the first approval, so the second approval is a no-op. |
| `data.requestedBy` field missing on old legacy-written notifications | The reply `fbSendNotification` call is guarded: `if (data?.requestedBy) { ... }`. |
